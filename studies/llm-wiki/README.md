# LLM Wiki

## 한 줄 요약

LLM Wiki는 LLM이 계속 읽고, 정리하고, 고치면서 유지하는 지식 베이스다.

## 왜 흥미로운가

LLM Wiki는 단순한 노트 폴더가 아니다.

중요한 점은 LLM이 작업 중에 배운 내용을 wiki에 반영하고, 다음 작업 때 그 wiki를 다시 읽는다는 것이다. 즉 지식이 대화 안에서 사라지지 않고, 파일로 누적된다.

## 출발점

이 아이디어는 Andrej Karpathy의 `llm-wiki.md` gist에서 출발했다.

## 핵심 흐름

```text
[새 자료 / 작업 / 대화]
   |
   v
[LLM이 오래 쓸 지식을 뽑음]
   |
   v
[LLM이 wiki를 업데이트]
   |
   v
[다음 LLM 세션이 wiki를 읽음]
   |
   v
[지식이 누적됨]
```

## RAG와 다른 점

```text
RAG
  - 질문할 때 원문 조각을 검색한다
  - 답은 그때그때 만든다
  - 정리된 기억이 자연스럽게 쌓이지 않는다

LLM Wiki
  - 새 자료를 넣을 때 정리한다
  - 정리된 wiki 문서를 계속 고친다
  - 다음 작업에서 다시 읽고 쓴다
```

## claude-automate brain과의 관계

우리의 예전 `claude-automate`에는 `.claude/brain` 개념이 있었다. 이것은 LLM Wiki와 매우 비슷하다.

다만 범위가 다르다.

```text
Project Brain
  - 한 프로젝트 안에 있음
  - 코드 구조, 패턴, 결정 기록을 저장
  - 그 프로젝트를 더 잘 고치기 위한 기억

LLM Wiki
  - 프로젝트 밖에 따로 있을 수 있음
  - 여러 프로젝트, 공부, 자료, 생각을 저장
  - 개인 또는 팀의 장기 지식 베이스에 가까움
```

`claude-automate`의 brain은 LLM Wiki 패턴을 프로젝트 단위로 좁힌 버전이라고 볼 수 있다.

## 어려운 점

LLM Wiki는 말로는 좋아 보이지만, 관리가 어렵다.

문제가 생길 수 있는 지점:

- raw 자료만 계속 쌓임
- 비슷한 페이지가 중복됨
- 오래된 판단이 남음
- 링크가 깨짐
- 출처가 불명확해짐
- LLM이 새 페이지를 너무 쉽게 만듦
- 사람이 봐도 뭐가 맞는지 헷갈림

## 현재 판단

처음부터 거대한 global wiki를 만들면 관리 비용이 너무 커질 수 있다.

그래서 처음에는 "모든 지식 저장소"가 아니라, 작은 `wiki-lab`으로 시작하는 편이 좋아 보인다.

```text
wiki-lab/
  inbox.md
  raw/
  wiki/
  index.md
  rules.md
```

핵심은 `inbox`를 먼저 두는 것이다. 아무 내용이나 바로 wiki에 넣지 않고, 재사용 가치가 있는 내용만 정리해서 올린다.

## 저장 기준 초안

저장한다:

- 나중에 다시 쓸 판단
- 반복해서 나오는 패턴
- 기술 선택 이유
- 좋은 설명 방식
- 실험 결과와 결론
- 내 생각이 바뀐 이유

저장하지 않는다:

- 그냥 흥미로운 링크
- 아직 읽지 않은 자료
- 일회성 대화
- 명령 출력
- 너무 일반적인 상식
- 출처 없는 주장

## 열려 있는 질문

- 이 wiki는 `frontier-lab` 안에 둘까, 별도 repo로 둘까?
- 어떤 내용은 자동 저장하고, 어떤 내용은 사람 확인을 받을까?
- project brain과 global wiki는 어떻게 연결할까?
- 오래된 문서는 어떻게 정리할까?
- LLM이 새 페이지를 만들 수 있는 기준은 무엇일까?

## 첫 실험

LLM Wiki 자체를 첫 실험 대상으로 삼는다.

목표:

- 지금 대화를 바탕으로 작은 문서를 만든다
- 2-3개의 wiki page로 나눠 본다
- LLM이 그 문서만 읽고 질문에 답하게 해본다
- 구조가 도움이 되는지, 관리가 귀찮은지 판단한다

## 지금 결론

LLM Wiki는 흥미롭다. 하지만 핵심은 "자동으로 지식이 쌓인다"가 아니다.

진짜 핵심은:

```text
LLM이 지식을 쌓게 하되,
망가지지 않게 하는 운영 규칙을 같이 설계하는 것
```

이다.
